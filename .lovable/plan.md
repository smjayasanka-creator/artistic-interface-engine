# Fixed Deposit (FD) Module

Build a full FD module inside the existing Mzizi Core shell — same sidebar, headers, cards, forms, and colours. All data in Lovable Cloud with RLS. Money stored as `numeric(18,2)` (LKR). Delivered in 5 phases so each stage is verifiable.

## Navigation & UI additions

Add a new sidebar section **Fixed Deposits** with:
- Register (list + filters)
- New Deposit
- Maturity Due (dashboard)

Admin gets a new tab **FD Products** alongside existing Products/Accounts tabs. All screens reuse `FormGrid`, `Card`, `Kpi`, `Badge`, existing button classes — no new design language.

## Data model (all under `public`, RLS on, GRANTs to authenticated + service_role)

- `fd_product` — code, name, min/max amount, payout options allowed (monthly, at_maturity), penalty_type (`rate_reduction` | `reprice_minus_margin`), penalty_value, wht_rate, auto_renewal_default, active
- `fd_rate_tier` — product_id, tenure_months, annual_rate, effective_from, effective_to (null = open)
- `fixed_deposit` — certificate_no (unique, `FD-YYYYMM-XXXXX`), company_id, branch_id, client_id, product_id, principal, rate_at_booking, tenure_months, payout_option, settlement_account (gl_account ref), maturity_instruction (`payout`|`renew_principal`|`renew_principal_interest`), value_date, maturity_date, status (`pending`|`active`|`matured`|`prematurely_closed`|`renewed`), parent_fd_id (for renewals), created_by, approved_by, approved_at, closed_at, close_reason
- `fd_interest_schedule` — deposit_id, seq, due_date, gross_interest, wht_amount, net_interest, paid (bool), paid_date
- `fd_accrual` — deposit_id, accrual_date, daily_amount, cumulative_amount, unique(deposit_id, accrual_date)
- `fd_transaction` — deposit_id, type (`opening`|`interest_payout`|`premature_closure`|`maturity_payout`|`renewal`), amount, txn_date, reference
- `fd_nominee` — deposit_id, name, nic, relationship, percentage
- `fd_settings` — company_id (unique), default_wht_rate, certificate_seq counter table for FD numbering

RLS: company-scoped via existing `current_company_id()` / `is_company_member`. Approver-only mutations use `has_role(auth.uid(),'admin')` (reuse existing `app_role`; treat `admin` and `branch_manager` as approvers, `loan_officer` as maker).

## Interest engine (server functions in `src/lib/fd.functions.ts`)

- **Rate lookup**: pick tier where `tenure_months` matches and `value_date` between `effective_from`/`effective_to`.
- **Monthly payout**: schedule row per anniversary month; gross = P × r/12; wht = gross × wht_rate; net = gross − wht.
- **At maturity**:
  - tenure ≤ 12 months: simple interest = P × r × months/12
  - tenure > 12: compound annually for full years, simple for trailing partial year; WHT applied at maturity on total interest.
- **Daily accrual**: actual/365, generated by daily cron server route `/api/public/hooks/fd-accrue` (pg_cron every day 00:15) — inserts one `fd_accrual` row per active deposit.
- **Interest payout cron**: same route processes any `fd_interest_schedule` rows whose `due_date <= today` and marks paid, creating an `fd_transaction`.

## Screens

1. **Admin → FD Products** — list, create/edit product, manage rate tiers (grid with tenure/rate/effective_from), configure penalty rule + WHT + auto-renewal.
2. **New Fixed Deposit** (`/fd/new`) — customer search (reuse client search), product/tenure select → rate auto-fills, principal (min/max validated), payout option, settlement account (existing GL account), maturity instruction, nominees (repeatable rows with % totalling 100), submit as **pending**.
3. **FD Register** (`/fd`) — filterable table (status, product, maturity range), summary Kpi cards (portfolio value, active count, weighted avg rate, interest paid MTD, maturing this month), CSV export, maturity ladder tab.
4. **FD Detail** (`/fd/$id`) — header with certificate, status badge; tabs: Overview, Schedule (monthly payout table), Accruals, Nominees, Transactions. Actions: **Approve** (pending → active, generates schedule + first accrual), **Premature Close** (maker → approver), **Renew** (on/after maturity), **Print Certificate** (printable view).
5. **Maturity Due** (`/fd/maturity`) — tabs 7/30/60 days, action to process (payout / renew per instruction).

## Premature closure logic

- Actual period in complete months + trailing days.
- Applicable rate: look up published rate for that shorter tenure at value_date, subtract `penalty_value` (rate_reduction mode) or apply reprice-minus-margin; clamp ≥ 0.
- Recompute total interest entitled at applicable rate; net WHT.
- If monthly payout, sum already-paid net interest, compute excess = paid − entitled_net, deduct from principal.
- Show breakdown modal before confirm; on approve create `premature_closure` transaction, cancel unpaid schedule rows, stop accruals.

## Maturity processing

- Cron scans deposits where `maturity_date <= today AND status='active'`.
- `payout`: create maturity_payout transaction (principal + final interest net of WHT), status → matured.
- `renew_principal`: create new `fixed_deposit` with same product/tenure at rate effective today, parent_fd_id set; status of old → renewed; payout residual interest.
- `renew_principal_interest`: new principal = old principal + net interest at maturity.

## Technical notes

- Server functions live in `src/lib/fd.functions.ts` using `requireSupabaseAuth`; cron endpoints under `src/routes/api/public/hooks/` using service role after apikey verification.
- Certificate numbering uses a Postgres function `next_fd_certificate_no(company_id)` with a `fd_number_seq` table (per company + YYYYMM) inside a transaction to guarantee uniqueness.
- Reuse existing `gl_account` for settlement account references (no new account table).
- Follow existing balanced-posting pattern via `journal_entry`/`posting` for interest payouts and maturity payments so the ledger stays consistent.
